# AI 与拓扑耦合机制

## 架构

```
用户输入 → 前端 → 服务器 → AI API → 前端 → 解析 [op] → 执行拓扑操作
```

---

## 三个耦合环节

### 1. 用户 → AI：拓扑数据注入

每次发消息，前端将当前拓扑状态一并发送，服务器拼接到 prompt 中：

```
当前拓扑（共 5 个设备，3 条连线）：
  设备：R1 [ID=r1] 类型=router IP=192.168.1.1
    已用端口: GE0/0/0, GE0/0/1
    可用端口: GE0/0/2, GE0/0/3
  ...

现有连线：
  R1 --GE0/0/0→ R2 [GE0/0/0]
```

AI 由此掌握拓扑全貌，可做整体规划。

### 2. AI → 拓扑：操作块解析

AI 回复中以 `[op]` 标记操作指令，前端解析执行：

```
[op] add: type=router, id=R5, x=300, y=200
[op] add_edge: from=R1, to=R5, src_port=GE0/0/2, tgt_port=GE0/0/1
```

### 3. 持久化

拓扑存储于服务器：`data/projects/<项目名>/topo.json`  
对话历史存储于：`data/projects/<项目名>/sessions/<会话ID>/messages.json`

---

## 优势

- **自然语言驱动**：无需手动拖拽，说句话即可完成设备/连线增删
- **上下文感知**：AI 掌握设备 IP、端口、连线状态，可整体规划
- **操作可追溯**：`[op]` 块用户可见，确认后再执行，避免误操作
- **会话记忆**：历史完整保存，新会话 AI 可延续之前的工作

---

## 劣势

| 问题 | 说明 |
|------|------|
| 强依赖输出格式 | `[op]` 格式偏差即解析失败，无纠错机制 |
| 快照式同步 | 手动改动后 AI 需下一轮消息才能感知 |
| 无事务性 | 多步操作中途失败不回滚，结果可能不一致 |
| AI 幻觉 | 可能建议操作不存在的设备 ID，静默失败 |
| 紧耦合 | AI 格式变化直接影响拓扑功能 |

---

## 改进方向

- **操作校验**：执行前验证，失败时让 AI 重试
- **事务执行**：全部验证通过后再统一执行
- **实时同步**：通过 WebSocket 将拓扑变化实时推送给 AI
- **失败反馈**：执行结果检查，失败时 AI 重新规划
